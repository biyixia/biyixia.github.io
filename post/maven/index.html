当我们对一个项目进行打包时，maven分别执行了

验证（validate） ==> 编译源码（compile） ==> 编译测试源码（test-compile） ==> 单元测试（test）==> 打包（package）

  1. 这个过程就是生命周期(lifecycle)。在生命周期的过程中会有很多的阶段(phase)（如clean由pre-clean、clean、post-clean组成），这些阶段按照既定的顺序执行来完成一个项目的构建。
  2. 在生命周期中，各个阶段的工作由插件来完成(plugin)，一个插件通常可以完成一个或者多个阶段的工作。每个阶段的工作对应插件中的一个目标(goal)。不同的插件结合起来，就完成了项目的构建。

lifecycle

Maven有三个内置的build lifecycle，分别是：clean、default、site

Clean

构建之前进行项目的清理（缓存，临时文件，编译文件等）

  1. pre-clean 执行一些需要在clean 之前完成的工作
  2. clean 移除所有上一次构建生成的文件
  3. post-clean 执行一些需要在clean 之后立即完成的工作

Default

负责项目的部署，构建的核心部分，编译，测试，打包，安装，部署等。

  1. compile 编译项目源码
  2. test-Compile 编译测试源代码
  3. test 使用合适的单元测试框架运行测试，这些测试代码不会打包或者部署
  4. package 接受编译好的代码，打包成可发布的格式，如 JAR
  5. install 将包安装至本地仓库，以让其他项目依赖
  6. deploy 将最终的包复制到远程仓库，以让其他开发人员项目共享或部署到服务器上运行

Site

负责项目文档的创建，生成项目报告，站点 ，发布站点

  1. pre-site 执行一些需要在生成站点文档之前完成的工作
  2. site 生成项目的站点文档
  3. site-deploy 将生成的站点文档部署到特定的服务器上

注意

  1. 当执行命令mvn [phase]时，[phase]及其所在build lifecycle中所有位于[phase]前的build phase都将被按序执行。（运行任何一个阶段的时候，它前面的所有阶段都会被运行 ）

  2. 在build phase上可以绑定plugin goal。若build phase上未绑定任何plugin goal，则该build phase将不会产生任何影响。若build phase上绑定了一个或多个plugin goal，则所有绑定的plugin goal都将被执行。

  3. 有两种方式可以为build phase绑定plugin goal：

    1. 在POM中设置packaging标签，其值可以被设为jar，war，ear，pom。这些packaging值将会为每个build lifecycle中对应的build phase绑定对应的plugin goal。（未指定packaging标签时，其将被默认设置为jar）

    2. 在POM中设置plugin标签，为某个<phase>绑定上该plugin下的某个或某些<goal>。

  4. 对于绑定了多个plugin goal的build phase，plugin goal的执行顺序为：先执行packaging中绑定的goal，后按序执行POM中绑定的goal。

POM(Project Object Mode)

Maven基于项目中的POM配置文件来对当前项目进行构建

标签 说明
modelVersion 固定值4.0.0
groupId 公司或组织的 id，即公司或组织域名的倒序
artifactId 项目或某个模块的名称，作为 Maven 工程的工程名
version 版本号
  1. groupId，artifactId，version构成了该项目的坐标，其唯一确定了一个project
  2. 对应Maven本地仓库根目录\【groupId】\【artifactId】\【version】\【artifactId】-【version】.jar

依赖配置补充

maven项目依赖生效分为三个时期，编译期、测试期、运行期

通过对<scope>合理的配置，可以控制当前依赖在不同时期编译期、测试期、运行期生效的关系，指定被依赖资源的依赖范围

可选配置有 compile、test、provided、runtime、system、import，若不指定则默认 compile

使用compile依赖范围

默认会使用该依赖范围，编译期、测试期、运行期都生效。

使用test依赖范围

只在测试期生效，典型的是JUnit依赖，只在编译测试代码（src/test）和运行测试的时候才需要

使用provided 依赖范围

只在编译期、测试期有效。表示当前依赖在运行时默认由某个子项目项目引用，运行期则不需要引用

上述工具项目中引用了spring-web依赖，但是当前项目同时引用了spring-boot-starter-web（包括spring-web）

既然运行时SpringBoot会提供spring-web，因此使用provided依赖范围在运行时不参与项目打包，避免重复引用依赖

使用runtime依赖范围

对于测试期、运行期有效

典型的例子是JDBC驱动实现，项目主代码的编译只需要JDK提供的JDBC接口，只有在执行测试或者运行项目的时候才需要实现上述接口的具体JDBC驱动。

使用system 依赖范围

和provided依赖范围一致（编译期、测试期），但需要指定systemPath标签，显示的指定依赖jar的路径。使用此依赖范围，表示的是从本地磁盘上找依赖，不是从maven仓库中解析了。使用此依赖的时候要注意，因为是从自己本地磁盘上找依赖，因此如果多个人在共同开发，比较麻烦，还要拷贝那个依赖jar到对方机器上的相同路径下。因此不建议使用这个依赖范围。

但是，system 范围的依赖不会从 maven 仓库下载，而是从本地文件系统获取，使用 system 范围的依赖时必须通过 元素显式地指定依赖文件的路径。

由于此类依赖不是通过 Maven 仓库解析的，而且往往与本机系统绑定，可能造成构建（构建的产物有：classes和artifact）的不可移植，因此应该谨慎使用。

导入依赖范围。该依赖对于上边说的三种classpath期都没有作用。该范围的依赖只在dependencyManagement标签才有效。它的作用是将目标pom的dependencyManagement标签中的依赖，合并到当前pom的dependencyManagement标签内，供引用项目中使用。由于import依赖的特殊性，一般看到import范围时，就会存在pom，即指向打包类型为pom的模块。比如下边这样：

\2. 依赖传递机制：通过该机制，Maven可根据项目的直接依赖，获取到其所有的间接依赖。

如当前项目A直接依赖于B和C，而B和C分别直接依赖于D和E。通过依赖传递机制，我们仅需在项目A中配置其直接依赖B和C，Maven会自动对间接依赖D和E进行获取。

runtime：在运行main文件夹中的代码时，该依赖不会被加入到classpath中（即对程序不可见）；在运行test文件夹中的代码时，该依赖会被加入到classpath中（即对程序可见）；在打包时，该依赖会被放入包中（即该应用在发布运行时，仍旧使用该依赖）。
