SpringSecurity

简介

Spring Security是一个能够为基于Spring的企业应用系统提供声明式（注解）的安全访问控制解决方案的安全框架

它提供了一组可以在Spring应用上下文中配置的Bean，充分利用了Spring IoC，DI（控制反转Inversion of Control ,DI:Dependency Injection 依赖注入）和AOP（面向切面编程）功能

为应用系统提供声明式的安全访问控制功能，减少了为企业系统安全控制编写大量重复代码的工作

基础概念

认证（登录）

认证能够保护系统的隐私数据与资源，让用户以合法的身份访问系统的资源

认证是判断一个用户的身份是否合法的过程

认证的场景

  1. 登录请求：在初次使用注册后，输入账号和密码登录微信的过程
  2. 非登录请求：用户去访问要求验证用户的身份信息的系统资源（url接口）时，身份合法方可继续访问，不合法则拒绝访问

常用认证方式

用户名密码登录，二维码登录，手机短信登录，指纹认证等方式

保持登录状态

为了避免用户通过认证后访问系统资源仍需重复认证，系统需要将用户的信息保存，从而保持用户的登录状态，常见的保持登录状态的方式有：session会话、token令牌，其中session常应用与java web项目中

方式 服务端 客户端
session认证成功后 在session中保存用户信息 在cookie中保存sesssion_id
token认证成功后 服务端生成token并在Redis中保存 在cookie或localStorage中保存token
方式 客户端请求 服务端
session校验 携带 session_id和cookie中的用户信息 验证sesion中是否存在该用户信息
token校验 携带 token 验证Redis中该token是否有效
对比 基于session的认证方式（Servlet规范定制） 基于token的认证方式
服务端存储位置 存储session，占用内存资源 不需要存储token
客户端存储方式 需要支持cookie 不限制存储方式

token令牌

鉴权 判断用户是否有访问某个资源的权限

根据用户的权限控制用户使用资源的过程

授权

为认证通过的用户授予访问指定资源的权限，从而对用户的隐私数据进行更详细的划分

授权是用户认证通过后根据用户的权限来控制用户访问资源的过程（有则访问，无则拒绝）

授权的数据模型（RBAC）

用户 角色 权限

Who对What(which)进行How操作

概念 说明
主体（Subject） 用户或程序，需要访问系统资源
资源（Resource） 功能资源（菜单、页面、按钮，每个功能对应一个url）、实体资源（资源类型和资源实例）
权限（Permission） 规定用户能否操作指定资源，如用户查询权限、用户添加权限
主体 主体和角色 角色 角色和权限 权限（包括资源） 资源
用户id、账号、密码、… 用户id、角色id、… 角色id、角色名称、… 角色 id、权限id、… 权限id、权限标识、权限名称、资源名称、资源访问地址、… 资源id、资源名称、访问地址、…

RBAC

把权限打包给角色，分配给用户

一般指基于角色的访问控制，至少需要五张表实现，是实施面向企业安全策略的一种有效的访问控制方式

基于角色的访问控制（Role-Based Access Control） 基于资源的访问控制（Resource-Based Access Control）
按角色进行授权 按资源（或权限）进行授权
当角色与权限关系变化时，要修改相关授权代码，系统可扩展性差 定义权限标识无需修改授权代码系统可扩展性强

JWT（Json Web Token）

可以按照指定方式生成的令牌，可以在互联网中进行传递，生成令牌具有一定规则

由三部分组成（头部，载荷，签名）

组成 说明
头部 指定jwt的算法、编码方式，由于是jwt的标准，我们无法修改
载荷 指定加密的数据，即用户自定义的数据
签名 指定对称、非对称加密的密匙，可以保护当前的token
