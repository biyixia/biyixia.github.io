Git

Git是分布式代码版本管理系统，其分为客户端和服务端两个部分

服务器端：大家共同存储的一个终端，可以自己搭建，也可以选择别的公司更好的的处理

本地机： 程序员自己的电脑，保存编写的代码

Git服务端的具体产品有GitHub（国外的喵咪）和Gitee（国内的码云）两款比较出名

常用术语

常用术语 解释
本地仓库（Repository） 保存了工程开发中的每一个迭代版本
远程仓库 远程保存推送的版本，使用github、码云等网站可以进行远程版本库的创建
工作区（Working Directory) 本地电脑上的工作目录文件夹
暂存区（Staging area） 暂存对工作区中文件的操作
索引（Index） 暂存区的别名
签入（Check in） 将新版本复制回仓库
签出（Check out） 从仓库中将文件的最新修订版本复制到工作区
提交（Commit） 将暂存区的文件提交到本地仓库
冲突（Conflict） 多人对同一文件的同一部分进行了修改，导致了冲突的发生
分支（Branch） 两个子版本，当前版本出现了两个不同分支
合并（Merge） 将指定的不同的版本合并成当前版本，合并过程中对于冲突的文件需要手动选择保留哪个文件
头（HEAD） 指向当前选择的分支
修订（Revision） 表示代码的一个版本状态。Git通过用SHA1 hash算法表示的ID来标识不同的版本
标记（Tags） 标记某个分支的某一个版本，方便检索和查看
推送（Push） 将当前的版本推送到远程仓库
拉取（Pull） 从远程仓库拉取文件，并合并入当前分支
add 添加新的文件（文件夹）到Git项目中，如果添加文件夹，该文件夹下所有文件将被包含
reset 选择commit前面版本重新开始编辑，通常是选择恢复到上一个编辑点
diff 找出两个文档或目录的不同
revert 恢复版本到某个commit前（即使push也能恢复，相当于过去的版本重新覆盖当前版本）
undo commit 撤销某个commit（push后无法撤销）
clone 从目标服务器下载整个用Git管理项目的
issues 问题
fork 分叉
rebase 重定
star 星

本地开发

文件状态

文件状态 说明 存储位置 去向
Untracked 未跟踪 仅工作区，未进行版本控制 可通过git add 将其加入暂存区
Staged 已暂存 工作区->暂存区 commit 加入本地仓库则变为Unmodified，修改则变为Modified
Modified 已修改 工作区、暂存区（内容不同） commit 将修改文件加入暂存区并上传至本地仓库，或checkout 丢弃修改, 返回unmodify状态（用暂存区的文件覆盖工作区文件）
Unmodified 未修改 工作区、本地仓库（内容相同） 修改则变为Modified，移出本地仓库则变为Untracked

  1. 在工作区新建一个文件 a.txt 。 此时文件状态为 Untracked
  2. 执行 git add a.txt 。 此时文件状态为 Staged
  3. 修改 a.txt 文件并保存，此时文件状态为 Modified，此时可进行两种操作：
    1. 执行 git add a.txt 。 此时文件状态又为 Staged
    2. 执行 git commit -m ‘提交a文件’ ，此时文件状态变为 Unmodified，此时可进行：
      1. 执行 git rm --cached a.txt 。此时工作区的a.txt变为Untracked状态

数据结构

某个结点表示该版本中的所有文件。箭头表示HEAD指针。

常用命令

常用命令 解释
git init 在当前目录下生成一个名为.git的文件夹，该文件夹即为版本库
git config 在将版本提交到版本库之前需要使用git config user.name xxx和git config user.email xxx设置好个人身份签名
git add xxx 将处理后的文件xxx的最新状态同步至暂存区中
git commit -m “XXX” 将暂存区中暂存的所有改动写进版本库中，形成一个新的版本。commit log表示对该版本的一个说明
git status 查看工作区（在工作区中进行了改动还未同步到暂存区的文件）、暂存区（在暂存区中还未写入版本库的文件）的状态
git log 显示版本库中，HEAD指针之前的所有版本的详细信息（space向下翻页、b向上翻页、q退出）
git log --oneline 显示版本库中，HEAD指针之前的所有版本的简略信息
git reflog 显示版本库中，HEAD指针之前及之后的所有版本的简略信息
git reset --hard xxx 将HEAD指针移动到版本哈希为xxx的版本处，并使工作区重置为该版本下的状态
git revert versionX…versionY 执行版本X到版本Y之间（不含X含Y）所有版本的逆向操作，使得这些版本的提交所造成的影响被撤销。对指定的每个版本进行撤销都会产生一次新的提交记录。（可添加–no-commit参数使得对所有版本的撤销操作全都保存在暂存区中，再通过commit命令一次性提交）
git diff 文件名 查看这个文件和之前文件的对比

远程多人协作

远程仓库地址：用于定位远程仓库。以github为例，其为每个仓库提供HTTPS协议和SSH协议的两种地址

常用命令

常用命令 解释
git remote add 【别名】【 远程仓库地址】 为远程仓库地址取别名
git remote -v 查看当前所有的远程仓库别名
git clone【 远程仓库地址】 将远程仓库完整下载下来成为本地仓库
git pull 【远程仓库地址别名】【 远程分支名】 将远程仓库中的指定分支拉取到本地，pull操作等同于fetch操作+merge操作
git fetch【远程仓库地址别名】/【 远程分支名】 获取远程仓库中的指定分支到本地
git push【远程仓库别名】【 本地分支名】 将本地仓库指定分支中HEAD指针所指的版本推送到远程仓库中，作为远程仓库的一次版本更新
  1. 推送到远程仓库的版本必须是基于远程仓库中最新版本所做的修改。否则，需要先 pull拉取最新的内容后再push推送
  2. 只有远程仓库所属团队的团队成员才能够执行push操作对该远程仓库进行更新

pull request

非远程仓库所属团队的团队成员需通过pull request的方式贡献代码

  1. 对他人的远程仓库进行fork。该操作将会得到一个他人远程仓库的完整副本，并且该远程仓库副本是属于自己的
  2. 将贡献的代码更新到fork出来的自己的远程仓库中
  3. 在fork出来的自己的远程仓库中执行pull request操作，请求原远程仓库所属团队成员进行代码审核，来将自己改动的内容合并进他们的远程仓库中

分支管理

  1. git支持分布式开发，集中式开发，分支式处理
  2. 以主分支为核心，创建子分支，利用子分支扩张内容，最后与主分支合并
  3. 从数据上来说呢，每个分支相当于独立的，我们可以在当前分支中进行回溯和查看版本（如:git reflog）
  4. 分支可以任意的切换
  5. 主分支master一般不处理具体的内容，主要负责调配调用子分支，子分支负责处理具体的内容

数据结构

（在分支X下仅能操作分支X自身的HEAD指针）

合并

fast forward

  1. 原分支（Master）的HEAD指针移动至其他分支（Branch）的HEAD指针处。
  2. 分支后，原分支未进行过提交。

three way merge

处于该种情况时，在原分支（Master）下对其他分支（Branch）进行的合并操作会使得原分支产生一次新的提交

合并时，git会比对版本F和版本G相对于它们的公共祖先版本C所做的改动：

  1. 版本F和版本G相对于版本C新增的文件，在版本H中都会存在（新增文件名相同文件某行内容不同也会发生冲突）
  2. 版本F和版本G相对于版本C减少的文件，在版本H中都不会存在
  3. 若两个分支都对同一个文件中的同一行进行了修改，则会发生冲突
  4. 若一个分支删除了另一个分支进行了修改的文件，则会发生冲突

变基

主分支发生了变化，主分支由其他的子分支合并了新的内容，另一个子分支也需要新增的这些内容，就可以让子分支变基(让主分支的新的内容重新分配给子分支)

先选择要变基的子分支，选中主分支右键选择Rebase即可，主分支中的内容就会重新分配给当前子分支。

情况一:主分支新建了一个子分支，都有内容的改变，推荐先让子分支合并到主分支中，子分支在变基，最后子分支在合并到主分支中。

情况二:分支不是由master创建的，而是由其他的子分支创建的，可以正常使用，分支结构线会有点乱

GitFlow工作流

分支 说明
主干分支master 仅负责更新正在运行的生产环境下的版本
Bug修理分支bugfix 负责更新master分支中出现紧急bug后，进行bug修复的版本。修复完毕后将并回master分支
预发布分支release 负责更新准备发布到主干分支的预发布版本，供一小部分发烧用户预先使用，来收集意见以及测试bug。若出现bug，一般直接在该分支下进行修复；若无问题，则并入主干分支
开发分支develop 负责更新开发过程中的版本，一般仅在该分支中更新短周期的开发。当开发完成到一定阶段后，将并入预发布分支，进行正式发布前的最后测试
功能分支feature 负责更新针对某一功能的需要进行长周期开发的版本，使得对不同功能的开发可以并行推进，并且不会影响到开发分支中短周期的开发工作。功能开发完成后，将并入开发分支

SSH 配置

若远程仓库地址使用的是SSH协议，则需要生成公钥（id_rsa.pub文件）

  1. 右键Git Bash Here打开终端窗口 ssh-keygen -t rsa -C "XXXXXXXXX@qq.com"
  2. 按照提示完成三次回车生成 ssh key
  3. cat ~/.ssh/id_rsa.pub 查看生成的钥匙信息
  4. 复制ssh公钥添加到Gitee的SSH公钥管理
  5. 回到终端窗口 ssh -T git@gitee.com Hi XXX! You've successfully authenticated, but Gitee.com does not provide shell access. 则添加成功

首次提交

新增文件交给Git

新建文件会提示add的操作，将新建文件加入到git的本地暂存区，让文件被git管理

提交新版本

选择需要提交到本地仓库的文件

commit 将处理好的代码文件提交到版本库中，形成一个新的版本

版本内容对比

  1. 选择提交版本
  2. 右侧在选择的版本中指定要对比的文件
  3. 点击 Show Diff 展示版本文件对比

推送至远程仓库

push 将git仓库中的内容推送到码云的远程服务器上

更新项目

若远程仓库与本地仓库版本不一致，需要先对本地仓库版本备份，拉取（更新）远程仓库的版本并合并（解决冲突）

冲突发生情况

本地仓库版本 远程仓库版本 更新项目
没有改变 改变 直接更新到本地仓库
改变 改变 可能发生冲突
/ 没有改变 无更新

如何避免冲突

  1. 错开推送代码到远程仓库的时间
  2. 在更新版本代码前，需先将远程仓库版本更新到本地仓库

撤销提交

undo commit

提交本地仓库版本后，若尚未推送至远程仓库，则可以撤销本次提交且删除记录

撤销提交不会将暂存区的改动撤销，还需要手动回滚

恢复提交

revert commit

提交本地仓库版本后，若已经推送至远程仓库，则只能恢复先前的版本（相当于重新提交先前做过的修改），留下操作记录

恢复提交会将暂存区的改动撤销

版本回溯

将版本回溯至指定版本，版本回溯会将暂存区的改动撤销

版本回溯会清空 指定回溯版本后的 、未推送的版本记录

创建分支

右键将在当前分支的头结点创建分支，创建后自动切换到新建的分支

合并主分支

将子分支合并到主分支中，主分支就拥有了主分支的所有内容

右键主分支，切换至主分支

右键新分支，将新分支提交版本合并到主分支

合并后会覆盖主分支中的内容
