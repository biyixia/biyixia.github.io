CAP原则

CAP 原则又称 CAP 定理，指的是在一个分布式系统中，

一致性（Consistency）

可用性（Availability）

分区容错性（Partition tolerance）（这个特性是不可避免的），这三个要素最多只能同时实现两点，不可能三者兼顾

要素 说明
一致性（C） 在分布式系统中的所有数据备份，在同一时刻是否同样的值。（等同于所有节点访问同一份最新的数据副本）
可用性（A） 在集群中一部分节点故障后，集群整体是否还能响应客户端的读写请求。（对数据更新具备高可用性）
分区容错性（P） 以实际效果而言，分区相当于对通信的时限要求。系统如果不能在时限内达成数据一致性，就意味着发生了分区的情况，必须就当前操作在C和A之间做出选择。

在分布式系统中网络会存在脑裂的问题，部分Server与整个集群失去节点联系，无法组成一个群体。只有在AP和CP选择一个平衡点

分布式特征

Nacos Zookeeper Eureka
数据一致性 注重数据的一致性 不是很注重数据的一致性！
服务可用性Paxos（多数派） 若集群中的master挂了，则 zk 集群整体不对外提供服务了，需要选一个新的出来（120s 注重服务的可用性，当 Eureka 集群只有一台活着，它就能对外提供服务

RestTemplate的API

T getForObject(url,T ResponseType,uriVariables)

  1. url：请求地址
  2. ResponseType：响应字节码类型
  3. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合
  4. 直接返回T类型的结果

ResponseEntity<T> getForEntity(url,T ResponseType,uriVariables)

  1. url：请求地址
  2. ResponseType：响应字节码类型
  3. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合
  4. 返回ResponseEntity<T>封装的结果

T postForObject(url,Object request, T ResponseType,uriVariables)

  1. url：请求地址
  2. request：请求体封装的参数
  3. ResponseType：响应字节码类型
  4. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合
  5. 直接返回T类型的结果

ResponseEntity<T> postForEntity(url,Object request, T ResponseType,uriVariables)

  1. url：请求地址
  2. request：请求体封装的参数
  3. ResponseType：响应字节码类型
  4. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合
  5. 返回ResponseEntity<T>封装的结果

void put(url,Object request, uriVariables)

  1. url：请求地址
  2. request：要修改的数据
  3. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合

void delete(url,uriVariables)

  1. url：请求地址
  2. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合

exchange(url,HttpMethod,HttpEntity,ResponseType,uriVarialbes)

  1. url：请求地址
  2. HttpMethod：请求方式
  3. HttpEntity：封装请求头（可用于权限校验，比如传递令牌，传递身份信息）或请求体（POST或PUT的请求体数据）
  4. ResponseType：响应字节码类型
  5. uriVariables：若地址栏占位符有参数，则代表地址栏占位符封装的Map集合
  6. 返回ResponseEntity<T>封装的结果

当需要发送请求时传递请求头信息或远程调用PUT、DELETE并有Boolean作为返回值时可使用
